# Security Lead Verdict

## 1. Password rotation: ESCALATE-TO-USER
Rotation is the single non-negotiable control here. The Railway DSNs in `scripts/migrate-cards.mjs` at `d823576` are already pushed to `origin/feat/card-details` on GitHub — they must be treated as fully compromised the moment the repo flips public (and, arguably, now, since any prior contributor or GitHub incident could already have copied them). Only the user holds Railway credentials, so the crew cannot perform rotation itself. Do not halt the crew — cleanup work (delete file, scrub history, license, collector, etc.) is parallelizable and valuable — but open-source publication MUST block on confirmed rotation. Flag this as a P0 manual follow-up in SUMMARY.md with explicit "do not make repo public until rotated" language, and require user sign-off before any visibility flip.

## 2. Git history scrub: HOLD
Force-pushing `git filter-repo` results to `main` and `feat/card-details` is the correct long-term hygiene step, but sequencing and safety matter. Scrubbing before rotation is security theater — the secrets are already on GitHub's servers, in any fork, in the GitHub search index, and in any cloned copy, so removing them from our tree does nothing to the attacker's copy. Scrubbing after rotation is defense-in-depth against future repo mirrors, forks, and casual readers of git log, and is worth doing — but it rewrites `main` history, invalidates every collaborator clone, and breaks any in-flight PR. That is a coordination decision (who is still working on this repo? any open PRs?) the Release Lead and user must own. Hold until (a) passwords rotated, (b) collaborator list confirmed, (c) force-push window scheduled.
Conditions: gated behind verdict 1 completion and Release Lead coordination signoff.

## 3. Delete `scripts/migrate-cards.mjs`: PROCEED
The file is dead one-off migration code — not referenced from `package.json`, `Dockerfile`, `railway.toml`, or any app source. Removing it from HEAD stops the bleeding at the working-tree level so the next clone (post-rotation, post-scrub, or even just a fresh reader) does not see live-looking credentials. This is independent of rotation and scrubbing: even without them, deleting from HEAD is strictly better than leaving it. Do it now as part of the normal cleanup commit.
Conditions: confirm `git grep "migrate-cards"` returns only the file itself and `.cleanup/` artifacts before deletion; no npm/bun script references exist.

## 4. Move `apps/web/public/collector.mjs` to build-time: HOLD
Grep confirms a real runtime dependency: `apps/web/app/setup/collector/page.tsx` instructs users to `curl -fsSL ${WEB_URL}/collector.mjs | node -`, so the file must be served from `/public/` at runtime. The current Dockerfile does NOT build the collector — it only runs `bun run build` in the web workspace and ships the pre-committed artifact. Moving to build-time therefore requires (a) adding a `bun --filter='./apps/collector' run build` step before the web build in the Dockerfile, (b) verifying the output path matches `apps/web/public/collector.mjs`, (c) adding to `.gitignore`, (d) confirming Railway's build context still produces the artifact. This is a real refactor with deploy-risk — if it breaks, the `/setup/collector` flow silently 404s for new users. Low security value (the artifact is benign public JS), moderate deploy risk. Defer to a follow-up PR; do not bundle into the open-source cleanup commit.
Conditions: if pursued later, require staging deploy verification of `/collector.mjs` 200-response before merge.
